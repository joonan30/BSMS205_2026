Chapter 24 · Part IV · Population Genetics
Welcome to Chapter twenty-four, Data Types for Alleles and Populations. This is the final chapter of Part Four, and it is a shift of gears. In the previous chapters we talked about the biology — allele frequencies, population structure, linkage, recombination, haplotypes. Today we ask: how do we actually represent all of that in a computer? How do we store it, share it, and work with it? The answer is a single file format that sits at the centre of modern genomics. It is called V C F.
Today's central question

How do you turn
a genome
into data?

Here is the framing question for the entire lecture. How do you turn a genome into data? When you sequence someone, you do not directly read off their genome as a string of letters. You get billions of short fragments. Those fragments have to be aligned, compared, and condensed into a list of places where this person differs from the reference. Then that list needs to be written down in a format that computers can read, that other scientists can use, that databases can search. What is the right way to do that? The answer took twenty years to settle on, and today it is universal.

The universal answer · VCF

  • Variant Call Format
  • Every variant caller outputs VCF
  • Every database stores VCF or a sibling
  • Every analysis tool accepts VCF
The universal language of genetic variation.
The universal answer is a file format called V C F, which stands for Variant Call Format. Every variant caller — the software that finds differences between a sequenced sample and the reference genome — outputs V C F files. Every major genetic database stores variants in V C F or a close sibling. Every analysis tool that works with variation expects V C F as input. It is the universal language of genetic variation. If you are going to work in human genetics at any scale, you will encounter V C F every day.

§ 1

From Sequencing
to Variant Calling

Let's walk through how a physical D N A sample becomes a V C F file.

The pipeline

  1. Extract DNA · sequence on Illumina → billions of short reads
  2. Align to reference (GRCh38) → BAM / CRAM file
  3. Variant call against reference → VCF file

Common tools: BWA · Bowtie2 · GATK · DeepVariant · FreeBayes

The pipeline is three stages. Stage one. You extract D N A from a sample and sequence it on a platform like Illumina. The output is millions or billions of short sequencing reads, each typically one hundred or one hundred fifty bases long. Stage two. You align those reads to a reference genome. For humans, the current standard is G R C h thirty-eight. Alignment is done by tools like B W A or Bowtie two. The output is a B A M file, or its more compressed version C R A M — a sorted list of every read and where it came from. Stage three. You compare the aligned reads to the reference at every position, and ask: does this sample differ from the reference here? That is variant calling. Tools like G A T K, DeepVariant, and FreeBayes perform this step. The output is a V C F file.

What the variant caller decides

  • Reference at chr1:12345 is A
  • Reads at this position show consistent T
  • Caller evaluates: read quality · mapping quality · strand bias · coverage
  • Decision: is this a real variant, or noise?
Let's zoom into what the variant caller actually does. Say the reference has an A at a particular chromosomal position — chromosome one, position twelve thousand three hundred forty-five. Your sample's reads at that position consistently show a T instead. Is that a real variant, or is it sequencing noise? The caller evaluates multiple factors — read quality scores, mapping quality, strand bias, coverage depth, how the reads align around the variant. It assigns a probability that the variant is real, and if the probability is high enough, it outputs the variant. Otherwise it flags the site as uncertain or skips it. Every variant in a V C F file has been through this probabilistic test.
§ 2

VCF File Structure

Now let's look at the structure of a V C F file itself.

A VCF is a plain text file

  • One line per variant
  • Each line: where the variant is, what changes, how confident
  • Tab-separated columns
  • Header lines start with #
A V C F file is surprisingly simple in concept. It is a plain text file. One line per variant. Each line specifies where the variant is located, what the reference allele is, what the alternate allele is, and how confident the caller is. Columns are tab-separated. Header lines at the top of the file start with a hash character. If you open a V C F file in a text editor, you can read it with your own eyes. There is nothing binary about it.

Nine core columns

ColumnMeaning
CHROMChromosome (chr1, chrX, ...)
POSPosition (1-based)
IDIdentifier (rsXXXX or .)
REFReference allele
ALTAlternate allele
QUALQuality score
FILTERPASS or failure reason
INFOSite-level metadata
FORMATPer-sample field definitions
Here are the nine core columns every V C F row contains. C H R O M, the chromosome. P O S, the position in one-based coordinates. I D, an optional identifier — often a d b S N P r s number if the variant is catalogued, or a dot if novel. R E F, the reference allele. A L T, the alternate allele observed in this sample. Q U A L, a quality score — higher means more confidence, thirty means ninety-nine point nine percent confidence. F I L T E R, either PASS or a reason for failure. I N F O, a field packed with site-level metadata. F O R M A T, which describes the structure of the per-sample columns that follow. These nine columns are the backbone of every V C F file ever written.

One line · read it out

chr1  1234567  rs12345  A  T  99.0  PASS  AC=3;AN=6;AF=0.5  GT:DP  0/1:32
  • chr1 pos 1,234,567 · ref A · alt T
  • Quality 99 · passes filters · rsID rs12345
  • 3 alternate alleles in 6 total (AF = 50%)
  • This sample is heterozygous · 32-read depth
Let's read out one concrete V C F line. Chromosome one, position one million two hundred thirty-four thousand five hundred sixty-seven. This variant has the identifier r s twelve thousand three hundred forty-five from d b S N P. The reference allele is A, the alternate is T. Quality is ninety-nine, very high. It passed all filters. In the I N F O field, the allele count is three, the allele number is six, the allele frequency is fifty percent. The per-sample data says the genotype is zero slash one — heterozygous — and the depth of sequencing at this position is thirty-two reads. That single line packs all the essential information you need about this variant and this sample.
§ 3

The INFO Field ·
Population Statistics

The I N F O field is where population-level summary statistics live. Let's dig into it.

Three fundamental fields

FieldMeaning
AC Allele CountNumber of alternate alleles observed
AN Allele NumberTotal alleles successfully called · ideally 2 × N
AF Allele FrequencyAC ÷ AN
The three fundamental I N F O fields are A C, A N, and A F. A C stands for allele count — how many copies of the alternate allele were observed across all samples in the file. A N stands for allele number — the total number of alleles successfully called at this site. In a perfect dataset, A N equals two times the number of samples, because humans are diploid. But sometimes sequencing fails for some individuals at some sites, and those genotypes are marked as missing. A F, allele frequency, is simply A C divided by A N. These three fields together give you the population-level summary.

A concrete example

  • Sequence 100 people → 200 alleles total
  • 10 alternate alleles observed

AC

10

AN

200

AF

0.05
Here is a concrete example. You have sequenced one hundred people. Diploidy means you have two hundred alleles in total. Suppose you detect the alternate allele ten times across all one hundred people. Then A C equals ten, A N equals two hundred, and A F equals ten divided by two hundred, which is zero point zero five, or five percent. That A F value tells you the variant is common enough to be well-sampled but rare enough to be interesting. These are the same numbers we computed by hand in Chapter twenty for the allele frequency formula — they are stored explicitly in every V C F.

Why these three numbers matter

  • Tell you how common or rare a variant is
  • Help decide pathogenic vs benign
  • Feed directly into GWAS, burden tests, population genetics
Why these three numbers matter. They tell you immediately how common or rare the variant is in your population. They are the starting point for deciding whether a variant is likely pathogenic — rare variants in disease genes often are — versus likely benign, which most common variants are. And they feed directly into every downstream analysis — genome-wide association studies, rare-variant burden tests, population-genetic statistics. Before you interpret any variant, check its A F.
§ 4

Genotypes ·
FORMAT and GT

Now let's turn to individual-level genotypes — the per-sample information.

The FORMAT column

  • Defines what information appears for each sample
  • Colon-separated list of field names
  • Example: GT:DP:AD
  • Each sample column follows the same template
The F O R M A T column defines what per-sample information appears in the columns that follow. It is a colon-separated list of field names. For example, F O R M A T might be G T colon D P colon A D. Every sample column then follows the same template — three colon-separated values corresponding to G T, D P, and A D in that order. This lets V C F carry any number of per-sample fields in a flexible way.

Genotype codes · the heart of VCF

CodeMeaning
0/0Homozygous reference — no variant
0/1Heterozygous — one ref, one alt
1/1Homozygous alternate — both carry variant
./.Missing — genotype not called
The G T field is the heart of V C F. Here are the four codes you will see constantly. Zero slash zero means homozygous reference — both alleles match the reference, the person does not carry the variant. Zero slash one means heterozygous — one reference allele and one alternate allele, the person is a carrier. One slash one means homozygous alternate — both alleles carry the variant. Dot slash dot means missing — the genotype could not be determined, usually because sequencing coverage was too low or read quality too poor at this site. These four codes describe the genotype of any diploid individual at any bi-allelic site.

Phased vs unphased

Unphased · /

  • Know which alleles are present
  • Don't know which came from mom vs dad
  • The default from variant calling

Phased · |

  • Know which allele is on which chromosome
  • Essential for haplotype work
  • Requires extra inference or family data
There is a subtle detail in genotype notation. A slash between the alleles — zero slash one — means the genotype is unphased. We know the person has one reference and one alternate allele, but we do not know which came from the mother and which from the father. A pipe symbol between the alleles — zero pipe one — means the genotype is phased. We know which allele sits on which chromosome. Phasing is essential for haplotype analysis — the topic of the previous chapter — because it tells us which variants are inherited together on the same chromosome. Phasing requires either extra computation or family data. By default, raw variant calling produces unphased genotypes.

Supporting fields · DP and AD

FieldMeaningExample
DPTotal read depthDP=32
ADAllelic depth (ref, alt)AD=16,16
  • 0/1 + AD=16,16 → balanced heterozygote · consistent call
  • 0/1 + AD=28,4 → suspicious · possible sequencing error
Beyond G T, two supporting fields are essential. D P is total read depth — how many sequencing reads covered this position in this sample. Higher depth gives you more confidence. A depth of five is marginal. Thirty or more is solid. A D is allelic depth, broken down by allele. If A D is sixteen comma sixteen, that means sixteen reads support the reference and sixteen support the alternate. For a heterozygote — a zero slash one — you would expect roughly fifty-fifty support, so sixteen and sixteen is consistent with the call. If instead you saw a zero slash one with A D of twenty-eight comma four, you would be suspicious — maybe the four alt reads are sequencing errors and the true genotype is homozygous reference.
§ 5

Multi-Sample VCF
and Joint Genotyping

The real power of V C F comes when you work with many samples together.

Multi-sample VCF (msVCF)

  • One row per variant site
  • One column per sample
  • INFO summarises population · each sample column gives individual genotype
  • Same file · population and individual views simultaneously
The multi-sample V C F, abbreviated m s V C F, is how we work with population data. Each variant site is one row. Each sample is one column. The I N F O field summarises population-level statistics like A C, A N, and A F. And each sample column gives that individual's genotype at that site. In a single file you have both the population view and the per-individual view. That is the fundamental data structure of modern human genetics.

Three samples · one variant

PersonGenotypeT alleles
Alice0/1 · heterozygous1
Bob1/1 · homozygous alt2
Carol0/0 · homozygous ref0

→ AC = 3 · AN = 6 · AF = 0.5

Let's trace through a concrete example with three people. At a single site on chromosome one, the reference is A and we have detected a T variant. Alice has genotype zero slash one — heterozygous. She contributes one T allele. Bob is one slash one — homozygous alternate. He contributes two T alleles. Carol is zero slash zero — homozygous reference. She contributes zero T alleles. Total: three T alleles out of three people times two alleles, equals six total. So A C is three, A N is six, A F is zero point five. The same count we did by hand in Chapter twenty is now encoded in the I N F O field of one V C F line.

Variant ≠ Genotype

A variant is a site where an alternate allele exists in the population.
A genotype is an individual's specific allele combination.

One variant · many genotypes.
Variant = population-level · genotype = individual-level.

Here is a distinction that trips people up. A variant and a genotype are different concepts. A variant is a site where the reference allele and some alternate allele coexist in the population. That is a population-level statement — it exists regardless of any individual. A genotype is an individual's specific allele combination at that site. Alice, Bob, and Carol share the same variant — they all live in a population where the T allele exists. But they have three different genotypes — zero slash one, one slash one, and zero slash zero. One variant, many genotypes. Always keep the distinction clear.

Joint genotyping

  • Call variants across all samples simultaneously
  • Even sites where most people are 0/0 get an explicit call
  • Prevents missing data from corrupting AC / AN / AF
  • Standard for gnomAD, UK Biobank, All of Us
There is an important technical detail about how multi-sample V C Fs are built. If you call variants on each sample independently and then merge the results, you have a problem. Sites that are variant in one person but reference in another may not be recorded at all for the reference person — they might look like missing data. That would corrupt your allele frequency calculations. The solution is called joint genotyping. You call variants across all samples simultaneously, looking at reads from everyone at once, and explicitly record the genotype for every sample at every site — including the zero slash zeros. This ensures complete, consistent data. Every large-scale genomic resource — gnomAD, U K Biobank, All of Us — is built from joint genotyping.

The attendance analogy

Without joint genotyping

  • Only write down who answered present
  • Don't know who was absent
  • Incomplete information

With joint genotyping

  • Full roster of who is present or absent
  • Complete information
  • Correct AF calculations
Here is the mental model. Imagine you are taking attendance in a class. Without joint genotyping, you are like a teacher who only writes down the names of students who answered present when called. You do not know who was actually absent versus who just was not called on. That is incomplete information. With joint genotyping, you have a full roster, and you explicitly mark every student as present or absent. Complete information. That is what joint genotyping does for variants. Every sample gets an explicit genotype at every site — even if that genotype is zero slash zero, meaning absent from the variant.
§ 6

Scaling ·
Dense vs Sparse

Now for the hard problem. What happens when your sample size goes from thousands to millions?

The scaling problem

  • UK Biobank · sequenced ~500,000 people
  • Dense VCF: one genotype entry per sample per site
  • For a rare variant: 499,990 samples are 0/0
  • Storing all those 0/0s is wasteful
Modern sequencing projects have exploded in scale. U K Biobank has sequenced about five hundred thousand people. The All of Us programme in the U S aims for a million. When you store a dense multi-sample V C F at that scale, each row has half a million genotype columns. And here is the inefficiency. For a rare variant — which is the majority of all variants — maybe ten people are one slash one or zero slash one carriers, and the other four hundred ninety-nine thousand nine hundred ninety are zero slash zero. Writing down half a million zero slash zeros on every row is enormously wasteful.

Sparse VCF

  • Store only non-reference genotypes
  • 0/0 entries are omitted, implied from absence
  • Population stats (AC, AN, AF) still in INFO — no information loss
  • For 500k samples · rare variant stores ~10 entries, not 500,000
The solution is called sparse V C F. The idea is simple. Store only the non-reference genotypes — the zero slash ones and one slash ones. Omit the zero slash zeros, and treat absence as implicit zero slash zero. For a rare variant in five hundred thousand samples, you store ten genotype entries instead of five hundred thousand. The population-level A C, A N, and A F are still stored in the I N F O field, so no information is lost. You just avoid the massive redundancy of the dense format.

Dense vs sparse · a comparison

AspectDenseSparse
StorageAll genotypes explicitOnly non-ref genotypes
File sizeVery large, superlinear growthLinear with carriers
Best for10s–1000s of samplesBiobank scale (100k+)
Computational feasibilityBreaks at biobank scaleEnables modern genomics
Here is the direct comparison. Dense V C F stores every genotype explicitly and scales badly — file size grows superlinearly with sample count. It works fine for small projects with tens to thousands of samples, but it breaks at biobank scale. Sparse V C F stores only non-reference genotypes and scales linearly with the number of variant carriers. It is the only computationally feasible format at biobank scale. When you hear about gnomAD, U K Biobank, or All of Us, you are hearing about projects that depend on sparse V C F or one of its cousins.
§ 7

The Future of VCF

V C F is two decades old. And it is straining under modern data volumes. Let's look at the future.

The fundamental challenge

  • Adding new samples means updating every existing row
  • File size grows superlinearly with sample count
  • gnomAD export in VCF = petabyte range
  • Doubling gnomAD → infeasible in standard VCF
Here is the fundamental challenge. When you add new samples to a multi-sample V C F, you do not just add new rows for new variants observed in those samples. You also need to update every existing row to record that the new samples do not carry the old variants. Those updates grow the data held by all samples simultaneously. File size grows superlinearly. Exporting gnomAD to standard V C F today is in the petabyte range — that is a thousand terabytes. Doubling the size of gnomAD, which will certainly happen, would make V C F export completely infeasible. The format is hitting a wall.

GA4GH Future of VCF Working Group

  • Global Alliance for Genomics and Health · stewards the VCF standard
  • Future of VCF group formed 2019, meets monthly
  • Interoperability with htsget · Beacon · CRAM / SAM / BAM
The V C F standard is maintained by the Global Alliance for Genomics and Health, abbreviated G A four G H. Their Future of V C F Working Group was formed in twenty nineteen and meets monthly. They are tasked with assessing scaling solutions and making recommendations. V C F also needs to stay interoperable with related standards like h t s get, for downloading variation data from specific regions, Beacon, for querying genomic variants, and the read-level formats like C R A M, S A M, and B A M.

Three approaches to scaling

  1. Specification tweaks · SAV, SVCR, spVCF · keep the text format, compress smartly
  2. New binary format · engineered for large scale and targeted queries
  3. API-based hybrid · query protocol returns data in any VCF form
Three broad approaches are on the table. First, specification tweaks — modifications to the V C F format itself that make it scale linearly or even sublinearly. Examples include Sparse Allele Vectors, or S A V, Scalable Variant Call Representation, or S V C R, and sparse Project V C F, or s p V C F. These keep the familiar text format but compress smartly. They are easy to adopt but depend on the community coalescing on one choice. Second, a completely new binary file format, engineered from scratch for large-scale data and targeted queries. Harder to adopt because it requires updating every library that reads or writes V C F. Third, an A P I-based hybrid — a query protocol over a backing store, returning data in whatever V C F flavour the client asks for. This third approach is considered the most promising path forward.

Why this scaling effort matters

  • Enables reuse of large population-genomics datasets
  • Supports virtual cohorts — combining studies to gain statistical power
  • Especially valuable for rare disease research
Why does this scaling effort matter? Because success enables the reuse of large population genomics datasets. The method of virtual cohorts — combining data from multiple separate studies to gain statistical power — particularly depends on V C F scaling. This matters most for rare disease research, where any single study has too few cases and the only route to discovery is pooling across many cohorts. The V C F scaling problem is therefore a central bottleneck for the next decade of medical genetics.
§ 8

Why VCF Matters

Let's end by asking, stepping back, why V C F matters at all.

Before VCF · the wild west

  • Every lab used its own format
  • Cross-study comparison was a nightmare
  • Custom converters for every pair of tools
  • Reproducibility was hard
Before V C F, variant data lived in the wild west. Every laboratory used its own format. Every analysis tool expected a different input. Comparing data across studies was a nightmare. You had to write custom converters for every pair of tools. Reproducibility was extremely difficult. This was the state of the field roughly until twenty ten.

After VCF · the universal standard

  • Directly compare variants across labs and technologies
  • gnomAD, ClinVar, dbSNP distribute in VCF
  • Clinical reporting · research · everything
  • The same format scales from one exome to one million genomes
After V C F, the landscape changed completely. A variant called in one laboratory can be directly compared to variants from another laboratory, even if they used different sequencing technologies or different variant callers. Major databases like gnomAD, ClinVar, and d b S N P distribute their data in V C F. Clinical diagnostic labs use V C F to report results to physicians. Every major analysis tool accepts V C F as input. And the same format that works for a single exome scales, with the right extensions, to millions of genomes. Universal standards have real value.
§ 9

Summary

Let's wrap up with the core takeaways.

What to take away

  • Pipeline: reads → BAM → VCF
  • VCF = plain text · nine core columns · one line per variant
  • INFO: AC · AN · AF — population-level statistics
  • FORMAT + GT: 0/0 · 0/1 · 1/1 · ./.
  • Joint genotyping = complete data across all samples
  • Sparse VCF = future-proof scaling to biobank size
Six things to remember. One. The pipeline from sequencing machine to V C F goes reads, then B A M or C R A M, then V C F. Two. V C F is a plain text file with nine core columns and one line per variant. Three. The I N F O field carries population-level statistics — A C, A N, and A F. Four. The F O R M A T field plus the G T field encode individual genotypes — zero slash zero, zero slash one, one slash one, and dot slash dot for missing. Five. Joint genotyping ensures complete and consistent data across all samples. And six. Sparse V C F formats are what make biobank-scale genomics computationally feasible.

The bigger picture

VCF is how genomes become data — how biological molecules become numbers that computers can analyse, compare, and share.
  • From a single A→T change
  • …to biobanks with millions of variants across hundreds of thousands of people
  • Encoded as 0/0, 0/1, 1/1
The bigger picture is this. V C F is how genomes become data — how biological molecules, physical D N A in cells, become numbers that computers can process, compare, and share. From a single A to T change in one person's genome, to biobank datasets containing millions of variants across hundreds of thousands of people. All encoded in the same compact digital language of zero slash zero, zero slash one, and one slash one. That is the achievement of V C F.
